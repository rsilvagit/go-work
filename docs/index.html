go-work

Agregador inteligente de vagas de emprego com filtragem avancada, protecoes anti-ban e notificacoes automaticas. Zero custo de operacao.

Go 1.25 Redis Docker GitHub Actions

Visao Geral

Problema

Acompanhar vagas em plataformas como Gupy exige acesso manual diario, buscas repetitivas e triagem de resultados irrelevantes. Nao existe uma forma nativa de receber apenas vagas novas, filtradas por stack, modelo de trabalho ou nivel de senioridade, direto no canal de comunicacao que voce ja usa.

Solucao

Uma CLI em Go que busca vagas na API da Gupy, aplica filtros inteligentes e envia notificacoes automaticas via Discord e Telegram. Execucao 100% serverless via GitHub Actions com cron diario gratuito.

Arquitetura e Design

Estrutura de Pacotes

cmd/go-work/ Composition Root — orquestracao e wiring
internal/scraper/ Strategy Pattern — fontes de dados (Gupy, extensivel)
internal/filter/ Engine de filtragem — MaxAge, tipo, modelo, nivel, regiao
internal/output/ Strategy Pattern — Console, Discord, Telegram
internal/model/ Dominio — Job (value object, zero dependencias)

Pacotes auxiliares: internal/cache/ (Redis) e internal/httpclient/ (HTTP resiliente). O grafo de dependencias e um DAG aciclico — nao ha imports circulares. Todo o codigo esta em internal/ para encapsulamento.

Modelo de Dominio

type Job struct {
    Title       string
    Company     string
    Location    string
    URL         string
    Description string
    Source      string
    PostedAt    time.Time
    JobType     string    // full-time, part-time, estagio, freelance
    WorkModel   string    // remoto, hibrido, presencial
    Level       string    // junior, pleno, senior
    Salary      string
}

Job e um value object — imutavel, sem ponteiros, sem estado compartilhado. Pode ser copiado e passado entre goroutines sem locks. Metodos: FullText() para busca e Key() para deduplicacao.

sync.WaitGroup controla o ciclo de vida. sync.Mutex protege o slice compartilhado. Deduplicacao e filtragem acontecem apos o Wait(), em single-threaded.

Fluxo de Fetch (por goroutine)

  • 1. Cache check — Se Redis disponivel e cache hit, retorna imediatamente
  • 2. Rate limit — Delay aleatorio 2-5s com jitter antes do request
  • 3. HTTP GET — Request com headers anti-ban e UA rotacionado
  • 4. Retry — Em caso de 429/503, backoff exponencial (2s, 4s, 8s)
  • 5. Parse — JSON da Gupy mapeado para []model.Job
  • 6. Filtro local — Localizacao filtrada no scraper se informada
  • 7. Cache set — Resultado salvo no Redis com TTL configuravel
  • 8. Merge — Append no slice compartilhado (protegido por Mutex)

Sistema Anti-Ban

O httpclient.Client implementa 5 camadas de protecao para evitar bloqueios:

CamadaImplementacaoObjetivo
User-Agent Rotation13 UAs reais (Chrome, Firefox, Edge, Safari)Evitar fingerprinting
Headers RealistasAccept, Accept-Language, Sec-Fetch-*, DNTSimular navegador real
Rate LimitingDelay aleatorio 2-5s entre requestsNao sobrecarregar o servidor
Retry + Backoff2s, 4s, 8s em 429/503 (max 3x)Respeitar rate limits
TLS 1.2+tls.Config{MinVersion: tls.VersionTLS12}Conexao segura

O jitter (delay aleatorio) e crucial — requests com intervalos exatos sao facilmente identificaveis como bots. A combinacao de delay variavel + UA rotation + headers realistas torna o trafego indistinguivel de um navegador real.

Decisoes Tecnicas (ADRs)

ADR-01: Go como linguagem principal
Contexto

Precisava de concorrencia nativa, binarios estaticos e simplicidade de deploy em CI.

Decisao

Go 1.25 — goroutines para paralelismo, binario estatico sem runtime, stdlib robusta.

Alternativas descartadas

Python (GIL), Node.js (deps pesadas), Rust (complexidade desproporcional).

ADR-02: API JSON ao inves de scraping HTML
Contexto

A Gupy expoe dados via paginas HTML e API JSON publica.

Decisao

Consumir a API JSON — dados estruturados, menor trafego, menos fragil que parsing HTML.

Trade-off

API nao documentada oficialmente — pode mudar sem aviso.

ADR-03: Cache Redis com graceful degradation
Contexto

Multiplas execucoes geravam chamadas repetidas. No CI, Redis nao esta disponivel.

Decisao

Redis opcional — se indisponivel, a aplicacao continua normalmente sem cache.

Beneficio

Funciona em qualquer ambiente. Zero dependencias obrigatorias alem do Go.

ADR-04: Filtro de 24h para evitar duplicatas
Contexto

Cron diario enviava as mesmas vagas repetidamente.

Decisao

Filtrar por PostedAt nas ultimas 24h — zero estado externo, zero complexidade.

Alternativas avaliadas

Redis set persistente, arquivo local, SQLite, Bloom Filter — todas requerem estado externo desnecessario.

Padroes de Codigo

Strategy Pattern — Scrapers e Writers

// Contrato para fontes de dados
type Scraper interface {
    Name() string
    Search(ctx context.Context, query, location string) ([]model.Job, error)
}

// Contrato para canais de saida
type ResultWriter interface {
    WriteJobs(jobs []model.Job) error
}

Adicionar uma nova fonte (LinkedIn) ou novo canal (Slack) requer apenas uma nova struct implementando a interface. Zero alteracao no core.

Options Pattern — Defaults Sensiveis

type Options struct {
    MaxAge    time.Duration  // default: 24h
    MinDelay  time.Duration  // default: 2s
    MaxDelay  time.Duration  // default: 5s
}
// Zero value = usa o default. Nenhuma config obrigatoria.

Composition Root

O main.go e o unico lugar que conhece todos os pacotes. Funciona como composition root: parseia config, cria dependencias, orquestra execucao. Nenhum pacote interno cria suas proprias dependencias — tudo e injetado.

Flag + Env Var Fallback

func envOrFlag(flagVal, envKey string) string {
    if flagVal != "" { return flagVal }
    return os.Getenv(envKey)
}
// Flag CLI > Env var > Default

Licoes Aprendidas

Competencias Demonstradas

Clean Architecture

Separacao em camadas com dependencias unidirecionais e DAG aciclico.

SOLID

Single Responsibility, Interface Segregation, Dependency Inversion aplicados em Go.

Go Idiomatico

Interfaces implicitas, error wrapping, context propagation, zero value util.

DevOps

CI/CD com GitHub Actions, Docker multi-stage, secret management, IaC.

Desafios Resolvidos

  • Rate Limiting da API — Resolvido com rate limit + jitter + backoff exponencial + UA rotation. O jitter e a chave.
  • Vagas duplicadas entre execucoes — Avaliadas 5 abordagens. Filtro por data (24h) foi a mais simples e eficaz.
  • Limites de mensagem — Chunking automatico com margem de seguranca para Discord (2000) e Telegram (4096).
  • MarkdownV2 do Telegram — Funcao escapeMarkdown() tratando 18 caracteres especiais.
